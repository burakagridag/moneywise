# ADR-005: Transaction Data Model

## Status
Accepted — 2026-04-29

## Context
Sprint 3 introduces the `Transactions` table as the central fact table of MoneyWise.
Every user-facing financial operation — expense recording, income entry, account-to-account
transfer — results in a row in this table. The decisions made here have permanent consequences
for balance correctness, query performance, and schema evolution.

Four design questions require explicit decisions:

1. What is the exact Drift table schema?
2. How is double-entry bookkeeping modelled with a single-table approach?
3. How are account balances computed — stored/cached or computed on read?
4. How does the schema migration from version 2 to 3 work?

---

## Decision

### 1. `Transactions` Drift Table Schema

```dart
// Drift table definition for financial transactions — data/local feature.
class Transactions extends Table {
  // Primary key — UUID v4 string, generated by uuid package.
  TextColumn get id => text()();

  // 'income' | 'expense' | 'transfer'
  TextColumn get type => text()();

  // Transaction date and time, stored as UTC.
  DateTimeColumn get date => dateTime()();

  // Always a positive value. Direction is determined by `type`.
  RealColumn get amount => real()();

  // ISO 4217, e.g. 'EUR', 'TRY'.
  TextColumn get currencyCode => text().withLength(min: 3, max: 3)();

  // Multiplier to convert `amount` to the app's main currency. Default 1.0.
  RealColumn get exchangeRate => real().withDefault(const Constant(1.0))();

  // Source account for income/expense; the FROM account for transfer.
  TextColumn get accountId => text().references(Accounts, #id)();

  // Destination account for transfer only; null for income/expense.
  TextColumn get toAccountId =>
      text().nullable().references(Accounts, #id)();

  // Expense/income category; null for transfers.
  TextColumn get categoryId =>
      text().nullable().references(Categories, #id)();

  // Sub-category (child category); null if no sub-category selected.
  TextColumn get subcategoryId =>
      text().nullable().references(Categories, #id)();

  // Short user-visible label shown in list rows.
  TextColumn get description => text().nullable()();

  // Longer freeform note stored alongside the transaction.
  TextColumn get note => text().nullable()();

  // JSON-encoded list of local file URIs for receipt photos (deferred to Sprint 4+).
  TextColumn get photoUris => text().nullable()();

  // FK to recurring_transactions template that spawned this instance (nullable).
  TextColumn get recurringId => text().nullable()();

  // FK to bookmark that was used to create this transaction (nullable).
  TextColumn get bookmarkId => text().nullable()();

  // When true, the transaction is excluded from all summary totals and charts.
  BoolColumn get isExcluded =>
      boolean().withDefault(const Constant(false))();

  DateTimeColumn get createdAt => dateTime().withDefault(currentDateAndTime)();
  DateTimeColumn get updatedAt => dateTime().withDefault(currentDateAndTime)();

  // Soft-delete flag — consistent with AccountGroups, Accounts, Categories.
  BoolColumn get isDeleted => boolean().withDefault(const Constant(false))();

  @override
  Set<Column> get primaryKey => {id};
}
```

**Note on `recurringId` and `bookmarkId`:** These are nullable text columns without
Drift-level foreign key constraints because `RecurringTransactions` and `Bookmarks`
tables are not added to the schema in Sprint 3. The constraints will be added when
those tables land (Sprint 4+). Until then, values are stored as plain strings.

**Indexes (must be created in the migration):**

```dart
// Inside MigrationStrategy.onUpgrade for from < 3:
await m.createIndex(
  Index('transactions_date_idx',
      'CREATE INDEX IF NOT EXISTS transactions_date_idx ON transactions (date DESC)'));
await m.createIndex(
  Index('transactions_account_date_idx',
      'CREATE INDEX IF NOT EXISTS transactions_account_date_idx ON transactions (account_id, date DESC)'));
await m.createIndex(
  Index('transactions_category_date_idx',
      'CREATE INDEX IF NOT EXISTS transactions_category_date_idx ON transactions (category_id, date DESC)'));
await m.createIndex(
  Index('transactions_type_date_idx',
      'CREATE INDEX IF NOT EXISTS transactions_type_date_idx ON transactions (type, date DESC)'));
```

### 2. Double-Entry Bookkeeping Approach

A **single-row per transaction** model is used. Transfers are not split into two rows.

| Transaction type | `accountId` | `toAccountId` | Effect |
|-----------------|-------------|---------------|--------|
| `income` | receiving account | null | balance of `accountId` increases |
| `expense` | paying account | null | balance of `accountId` decreases |
| `transfer` | source account | destination account | `accountId` decreases; `toAccountId` increases |

This is consistent with SPEC §7.2 and the reference app (Money Manager / Realbyte).
The balance formula (Section 3 below) handles all three types with a single SQL
expression, so no trigger or application-level split is needed.

**Invariants enforced at the repository layer (not DB constraints):**

- `amount > 0` always.
- For `type = 'transfer'`: `toAccountId != null` and `toAccountId != accountId`.
- For `type = 'income'` or `'expense'`: `toAccountId == null`.
- `categoryId` is required for income/expense; it is null for transfers.

### 3. Balance Calculation Strategy — Computed on Read

**Decision: balances are computed dynamically via a Drift reactive query. No cached
column is stored on the `accounts` table.**

The balance formula per SPEC §7.1:

```sql
balance =
  accounts.initial_balance
  + COALESCE(SUM(t.amount) FILTER (WHERE t.account_id = a.id   AND t.type = 'income'),  0)
  - COALESCE(SUM(t.amount) FILTER (WHERE t.account_id = a.id   AND t.type = 'expense'), 0)
  - COALESCE(SUM(t.amount) FILTER (WHERE t.account_id = a.id   AND t.type = 'transfer'),0)
  + COALESCE(SUM(t.amount) FILTER (WHERE t.to_account_id = a.id AND t.type = 'transfer'),0)
```

Only rows where `is_deleted = 0` and `is_excluded = 0` are included.

**Justification for computed-on-read vs. cached:**

| Criterion | Computed on read | Cached column |
|-----------|-----------------|---------------|
| Correctness on edit/delete | Automatic — no sync needed | Requires delta update on every write; off-by-one bugs are common |
| Correctness on import/restore | Automatic | Full recalculation pass required after import |
| Read performance | O(n transactions per account) — acceptable with indexes; reactive Drift streams batch updates | O(1) read but O(1) write overhead everywhere |
| Test complexity | One query to test | Two paths to test (write path + read cache) |
| Schema complexity | No extra column | Extra column + migration |

At Phase 1 scale (< 50 000 transactions per account), the indexed SUM query completes
in under 2 ms on a mid-range device. A Drift `Stream` emits a new value whenever
the `transactions` table changes, so the UI stays reactive without polling or manual
invalidation.

The SPEC §7.1 mention of "Drift trigger with cache" is acknowledged but deferred:
if profiling in Sprint 4+ reveals measurable latency (> 16 ms per balance read),
a `account_balance_cache` Drift view or a Drift trigger updating a denormalised
column will be introduced under a new ADR.

### 4. Schema Migration — schemaVersion 3

`database.dart` changes:

- `schemaVersion` increments from `2` to `3`.
- `Transactions` is added to the `@DriftDatabase(tables: [...])` annotation.
- `TransactionDao` is wired as a `late final` field.

`onUpgrade` handler for `from == 2, to == 3`:

```dart
onUpgrade: (m, from, to) async {
  // from=1 path preserved from Sprint 2.
  if (from < 2) {
    await m.createTable(accountGroups);
    await m.createTable(accounts);
    await m.createTable(categories);
    await _seedDefaultData();
  }
  // Sprint 3: add transactions table + indexes.
  if (from < 3) {
    await m.createTable(transactions);
    await m.createIndex(/* transactions_date_idx — see Section 1 */);
    await m.createIndex(/* transactions_account_date_idx */);
    await m.createIndex(/* transactions_category_date_idx */);
    await m.createIndex(/* transactions_type_date_idx */);
  }
},
```

**Important:** The `from < 3` block must run independently of `from < 2`.
Chaining with `else if` would prevent a user upgrading from version 1 directly to
version 3 from getting the transactions table.

**Note:** If SQLCipher re-keying (ADR-006) runs before `onUpgrade`, the migration
executes against an already-encrypted database. The `MigrationStrategy` sees a
normal SQLite connection regardless of encryption; no special handling is required.

---

### 5. Soft-Delete Consistency with Sprint 2

All Sprint 2 tables (`AccountGroups`, `Accounts`, `Categories`) use an `isDeleted`
boolean column. `Transactions` follows the same pattern:

- Deletes set `isDeleted = true` and `updatedAt = now()`. No physical row removal.
- All DAO queries filter `WHERE is_deleted = 0` by default.
- The balance formula excludes `is_deleted = 1` rows.
- QA verifies that soft-deleted transactions do not appear in any list, chart,
  or balance calculation.

Hard deletes are not exposed in the UI in Phase 1. A future "purge old data" screen
(Phase 3+) may physically remove rows after a configurable retention period.

## Consequences

### Positive
- Correct balances without any cache-invalidation logic.
- Transfer as a single row eliminates duplicate-entry bugs.
- Migration path is additive (no column renames or drops).
- Soft-delete pattern is consistent across all Sprint 2/3 tables.

### Negative
- Balance query touches all transactions for an account on every stream emission.
  Mitigated by four indexes; revisit with a cache ADR if profiling warrants.
- `recurringId` and `bookmarkId` lack DB-level FK constraints until Sprint 4.

## Alternatives Rejected

- **Two rows per transfer (split entry):** More flexible for some reporting queries
  but introduces the risk of orphaned half-entries on interrupted writes. Single-row
  with formula is simpler and correct.
- **Cached balance column:** Reduces read cost but introduces a synchronisation
  invariant that every write path must maintain. Error-prone and harder to test.
- **Drift view for balance:** Acceptable future path; premature optimisation now.
